PARTE 1 — INVENTARIO ACTUAL DE TESTS
================================================================================

Las 193 pruebas existentes se organizan en tres tipos:

  Tipo A — Tests unitarios de servicio (@ExtendWith(MockitoExtension.class))
    Todas las dependencias son mocks (@Mock). No hay Spring, BD ni HTTP.
    Verifica la lógica de negocio del servicio en aislamiento total.

  Tipo B — Tests de integración de controlador (@WebMvcTest + @AutoConfigureMockMvc(addFilters=false))
    Levanta solo la capa web de Spring MVC. El servicio es @MockBean.
    Verifica que los endpoints responden con los HTTP status correctos,
    que @Valid funciona, y que la serialización JSON es correcta.
    No hay BD, no hay Spring Security activo (filtros deshabilitados).

  Tipo C — Tests @SpringBootTest con contexto completo y BD real
    Solo existe para SaleOrderConcurrencyTest. Levanta la aplicación
    completa en un puerto aleatorio y envía HTTP requests reales.
    Verifica Optimistic Locking de Hibernate contra PostgreSQL real.

1.1  core/security — JwtUtilsTest (4 tests — Tipo A)
────────────────────────────────────────────────────────────────────────────────

CLASE BAJO PRUEBA: JwtUtils
TECNOLOGÍA: @ExtendWith(MockitoExtension.class) — sin Spring

CASOS DE PRUEBA:
  1. shouldGenerateValidToken
     Genera un token y verifica que extractUsername devuelve el username correcto
     y que validateToken retorna true.

  2. shouldExtractCorrectUsername
     Genera un token con username específico y verifica la extracción del claim 'sub'.

  3. shouldReturnFalseForInvalidOrManipulatedToken
     Verifica que un token con firma manipulada retorna false en validateToken.

  4. (setUp — no es test, prepara el entorno)

ANÁLISIS:
  Los 3 casos cubren los caminos críticos de JWT: generación, extracción y validación
  de firma. El caso de token expirado NO está cubierto — requeriría un token con
  expiración en milisegundos o un reloj virtual (Clock mockeable), lo cual es posible
  pero no implementado.

JUSTIFICACIÓN:
  JwtUtils es el componente de seguridad más crítico del sistema. Un error aquí
  podría permitir acceso no autorizado o bloquear usuarios legítimos.

CRITERIOS DE ÉXITO ACTUALES:
  ✓ Token generado con claims correctos
  ✓ Username extraído correctamente del payload
  ✓ Token manipulado rechazado

GAPS:
  ✗ Token expirado no se verifica en tests automatizados
  ✗ Token con issuer inválido no se prueba
  ✗ Token con claims faltantes (ej. sin 'sub') no se prueba

  ✓ StockMovement generado con tipo, quantity y reason correctos

GAPS:
  ✗ MapStruct real no ejecutado — supplier/category mapping no verificado
  ✗ reservedStock con valor > 0 no es un caso de @BeforeEach — podría
    agregar un test específico de OUT cuando reservedStock > 0

1.17 sales — SaleOrderConcurrencyTest (3 tests — Tipo C @SpringBootTest)
────────────────────────────────────────────────────────────────────────────────

CLASE BAJO PRUEBA: Stack completo (HTTP → Spring Security → Controller →
                   Service → Hibernate → PostgreSQL)
TECNOLOGÍA: @SpringBootTest(webEnvironment=RANDOM_PORT) + TestRestTemplate
            + CountDownLatch para sincronización de threads

MECANISMO DE CONCURRENCIA PROBADO:
  @Version en Product.java — Optimistic Locking de Hibernate.
  saveAndFlush() en approveOrder() para que la verificación de versión
  ocurra dentro del try-catch del servicio, convirtiendo
  ObjectOptimisticLockingFailureException al mensaje de negocio:
  "Stock modificado concurrentemente. Intente nuevamente."

CASOS DE PRUEBA:

  Test 1 — concurrentApprove_stockInsuficienteParaAmbos_soloUnoAprobado:
    Setup: producto con currentStock=8, dos órdenes de 8 unidades cada una.
    Acción: 2 threads disparan PATCH /approve simultáneamente.
    Verificación:
      - Exactamente 1 respuesta HTTP 200 (APPROVED)
      - Exactamente 1 respuesta HTTP 500
      - El body del 500 contiene "concurrentemente" O "insuficiente"
        (ambos son mensajes de negocio válidos según el timing de la carrera)
      - product.reservedStock == 8 (no 16 — sin double-booking)

  Test 2 — concurrentApprove_stockSuficienteParaAmbos_ambosAprobados:
    Setup: producto con currentStock=20, dos órdenes de 8 unidades.
    Acción: 2 threads simultáneos.
    Verificación:
      - Las 2 aprobaciones deben tener éxito (con reintentos si OOL)
      - product.reservedStock == 16
    Nota: si hay colisión de versión con stock suficiente, la perdedora
    sigue en PENDING y se reintenta — esto simula el retry de producción.

  Test 3 — concurrentApprove_altoNivelDeContencion:
    Setup: producto con currentStock=15, cinco órdenes de 5 unidades.
    Solo 3 pueden aprobarse (3×5=15). Las otras 2 deben ser rechazadas.
    Verificación:
      - reservedStock <= currentStock siempre (invariante crítica)
      - Todos los 500 tienen mensaje de negocio reconocible
      - Entre 1 y 3 aprobaciones exitosas (OOL puede rechazar algunas válidas)

ANÁLISIS PROFUNDO:
  Este es el único test del proyecto que ejercita Hibernate + PostgreSQL reales.
  Documenta un hallazgo importante: save() simple no garantiza que
  ObjectOptimisticLockingFailureException se capture dentro del try-catch
  porque el flush ocurre al commit de la transacción (fuera del bloque try).
  saveAndFlush() fuerza el SQL UPDATE inmediatamente, capturando la excepción
  en el lugar correcto.

CRITERIOS DE ÉXITO:
  ✓ Nunca se reserva más stock del disponible (invariante fundamental)
  ✓ El mensaje de rechazo es de negocio, no un mensaje interno de Hibernate
  ✓ Con stock suficiente, ambas aprobaciones eventualmente tienen éxito
  ✓ Alta contención (5 threads): reservedStock siempre <= currentStock

Los tests E2E con curl constituyen un tipo de prueba distinto a los Tipos A, B y C
documentados en la Parte 1. Se denominan aquí Tipo E (End-to-End):

  Tipo E — Tests E2E con curl / scripts bash

    Los scripts envían requests HTTP reales al servidor usando curl.
    El resultado se persiste en PostgreSQL real.
    No forman parte del pipeline de Maven (./mvnw test no los ejecuta).
    Son manuales en su ejecución pero automatizados en su verificación.

IMPORTANCIA HISTÓRICA EN ESTE PROYECTO:
  Los tests E2E con curl fueron los que detectaron los tres bugs críticos
  que los 118 tests con Mockito no encontraron, y que están documentados
  en CLAUDE.md como "Lecciones aprendidas":

  Bug 1 — supplier_id NULL en BD al crear producto
    Causa: ProductServiceImpl no llamaba resolveSupplier() — el campo supplier_id
    llegaba NULL a PostgreSQL violando el constraint NOT NULL.
    Los mocks aceptaron el guardado sin validar el campo.

  Bug 2 — supplierId null en el response aunque estaba en BD
    Causa: ProductMapper.toResponseDTO() no tenía @Mapping(source="supplier.id")
    Los mocks devolvían el DTO configurado en @BeforeEach — nunca ejecutaron MapStruct.

  Bug 3 — approvedBy/receivedBy/cancelledBy nunca persisten en BD
    Causa: updatable=false en los campos de transición de PurchaseOrder.
    Hibernate los excluía del UPDATE statement.
    Los mocks devolvían el mismo objeto en memoria donde el campo ya estaba seteado.

────────────────────────────────────────────────────────────────────────────────
1-E.4  Limitaciones de los tests E2E actuales
────────────────────────────────────────────────────────────────────────────────

LIMITACIÓN 1 — No forman parte del pipeline de Maven
  ./mvnw test NO ejecuta los scripts curl. Son pruebas manuales que requieren:
    a) La aplicación levantada
    b) Ejecutar el script manualmente
  Impacto: Un desarrollador que solo ejecute ./mvnw test puede tener una
  regresión en funcionalidad que los scripts detectarían.

LIMITACIÓN 2 — No son idempotentes sin limpieza previa
  Si se ejecutan dos veces sin truncar las tablas, los tests de creación
  fallan por nombres/SKUs/RFCs duplicados. Se requiere:
    PGPASSWORD=... psql -c "TRUNCATE TABLE ... CASCADE"
  antes de cada ejecución.
  Impacto: Dificultan la ejecución frecuente y la integración en CI/CD.

LIMITACIÓN 3 — Variable TOTAL colisiona en run_sales_e2e.sh
  El script bash usa $TOTAL tanto como contador de tests como para capturar
  totalAmount de las órdenes (números decimales como 7500.0).
  La aritmética entera de bash falla con números decimales, corrompiendo
  el contador y el display del resumen final.
  Impacto: El resumen muestra valores incorrectos, aunque los resultados
  individuales (✓/✗) son correctos.

LIMITACIÓN 4 — Cobertura de casos de error limitada
  Los scripts E2E cubren principalmente el happy path y los errores de negocio
  más importantes. Casos como:
    - Body JSON malformado (no es JSON válido)
    - Campos de tipo incorrecto (string donde se espera número)
    - URL con caracteres especiales
  No están en los scripts.

LIMITACIÓN 5 — No verifican cuerpo completo de todos los responses
  Algunos tests solo verifican el HTTP status code, no todos los campos
  del response body. Ejemplo: T07 verifica que la lista tiene 3 categorías,
  pero no inspecciona todos los campos de cada categoría.

NECESIDADES A RESOLVER:
  □ Integrar los scripts en un Makefile o en ./mvnw con un perfil e2e
    para que puedan ejecutarse con un solo comando
  □ Agregar idempotencia mediante sufijos únicos por ejecución
    (como hace SaleOrderConcurrencyTest con String suffix = timestamp)
  □ Corregir la colisión de variable TOTAL en run_sales_e2e.sh
  □ Convertir gradualmente los scripts curl en tests @SpringBootTest
    para que queden en el pipeline de Maven y sean parte de la suite
    ejecutable con ./mvnw test
  □ Agregar verificación de body completo en los tests más críticos
    (auditoría, campos de transición, unitCost)

────────────────────────────────────────────────────────────────────────────────
2.1  MEDICIÓN DE COBERTURA — JaCoCo
────────────────────────────────────────────────────────────────────────────────

ESTADO ACTUAL: No configurado. Sin métricas de cobertura.

QUÉ ES:
  JaCoCo (Java Code Coverage) es la herramienta estándar de la industria para
  Java. Instrumenta el bytecode y genera reportes de líneas, ramas y métodos
  ejecutados durante los tests. Se integra con Maven como plugin.

POR QUÉ ES NECESARIO:
  Sin JaCoCo no se puede saber:
  - Qué porcentaje del código se ejecuta en los tests
  - Qué ramas condicionales nunca se ejercitan
  - Qué métodos privados nunca son alcanzados indirectamente

  Los 193 tests actuales parecen completos, pero sin métricas no hay certeza.
  Ejemplo: la lógica de generación de números OV-YYYY-NNNN en caso de colisión
  (bucle do-while anti-colisión) probablemente nunca se ejercita en los tests
  porque el mock de findByOrderNumber siempre retorna Optional.empty().

ESTÁNDAR DE INDUSTRIA:
  - Mínimo aceptable: 60% de cobertura de líneas
  - Recomendado para sistemas críticos: 70-80%
  - Objetivo para módulos de seguridad (auth, JWT): 90%+
  - El número no es un fin — es una señal de alerta para código sin cobertura

IMPLEMENTACIÓN REQUERIDA:
  En pom.xml (dentro de <build><plugins>):

  <plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.11</version>
    <executions>
      <execution>
        <goals><goal>prepare-agent</goal></goals>
      </execution>
      <execution>
        <id>report</id>
        <phase>test</phase>
        <goals><goal>report</goal></goals>
      </execution>
      <execution>
        <id>check</id>
        <goals><goal>check</goal></goals>
        <configuration>
          <rules>
            <rule>
              <element>PACKAGE</element>
              <limits>
                <limit>
                  <counter>LINE</counter>
                  <value>COVEREDRATIO</value>
                  <minimum>0.70</minimum>
                </limit>
              </limits>
            </rule>
          </rules>
        </configuration>
      </execution>
    </executions>
  </plugin>

  Generará reporte en: target/site/jacoco/index.html

CRITERIO DE ÉXITO:
  ✓ Reporte HTML disponible en target/site/jacoco/
  ✓ Cobertura de líneas >= 70% en todos los paquetes de módulos
  ✓ Cobertura de ramas >= 60% en servicios con lógica condicional compleja
  ✓ Maven falla el build si la cobertura cae por debajo del umbral configurado

PRIORIDAD: Alta — debe implementarse antes de poder evaluar el resto de brechas

2.2  TESTS DE SEGURIDAD
────────────────────────────────────────────────────────────────────────────────

ESTADO ACTUAL: No existen. Todos los tests de controlador usan
               @AutoConfigureMockMvc(addFilters=false) que deshabilita
               Spring Security completamente.

QUÉ FALTA PROBAR:

  A) Rutas protegidas rechazan requests sin JWT:
     GET /api/v1/inventory/products/low-stock sin Authorization header → 403
     POST /api/v1/sales/orders sin Authorization header → 403

  B) Rutas públicas aceptan requests sin JWT:
     POST /api/v1/auth/register sin Authorization header → 201 (no 403)
     POST /api/v1/auth/login sin Authorization header → 200 (no 403)

  C) JWT inválido rechazado:
     GET /api/v1/inventory/products/low-stock con token manipulado → 403
     GET /api/v1/purchases/orders/status/PENDING con token expirado → 403

  D) JWT válido de usuario eliminado:
     Usuario creado → token generado → usuario borrado de BD →
     request con ese token → qué sucede (actualmente lanza RuntimeException
     en resolveAuthenticatedUser() porque no encuentra el username)

IMPLEMENTACIÓN REQUERIDA:
  Se necesita una clase nueva de test por módulo, usando @WebMvcTest SIN
  deshabilitar los filtros:

  @WebMvcTest(ProductController.class)
  // Sin @AutoConfigureMockMvc(addFilters = false)
  class ProductSecurityTest {

    @Autowired MockMvc mockMvc;
    @MockBean ProductService productService;
    @MockBean JwtUtils jwtUtils;

    @Test
    void getById_sinToken_retorna403() throws Exception {
        mockMvc.perform(get("/api/v1/inventory/products/1"))
               .andExpect(status().isForbidden());
    }

    @Test
    void getById_tokenInvalido_retorna403() throws Exception {
        when(jwtUtils.extractUsername(any())).thenThrow(new RuntimeException("invalid"));
        mockMvc.perform(get("/api/v1/inventory/products/1")
               .header("Authorization", "Bearer token.invalido.abc"))
               .andExpect(status().isForbidden());
    }

    @Test
    void getById_tokenValido_retorna200() throws Exception {
        when(jwtUtils.extractUsername("valid.token")).thenReturn("tester01");
        when(jwtUtils.validateToken("valid.token")).thenReturn(true);
        when(productService.getById(1L)).thenReturn(new ProductResponseDTO());

        mockMvc.perform(get("/api/v1/inventory/products/1")
               .header("Authorization", "Bearer valid.token"))
               .andExpect(status().isOk());
    }
  }

CLASES DE TEST REQUERIDAS (una por controlador con endpoints protegidos):
  SecurityTest para: UserController, CategoryController, ProductController,
                     SupplierController, PurchaseOrderController,
                     ClientController, SaleOrderController, ReservationController

CRITERIO DE ÉXITO:
  ✓ Todas las rutas protegidas retornan 403 sin JWT
  ✓ Todas las rutas públicas (/auth/**) funcionan sin JWT
  ✓ JWT con firma inválida rechazado con 403
  ✓ JWT expirado rechazado con 403
  ✓ JWT válido permite acceso a rutas protegidas

PRIORIDAD: Muy alta — es el único gap considerado bloqueante para producción.
           Un cambio en SecurityConfig podría romper el acceso sin que ningún

────────────────────────────────────────────────────────────────────────────────
2.5  PERSPECTIVA DE LOS 15 FACTORES (15-Factor App)
────────────────────────────────────────────────────────────────────────────────

Los 15-Factor App son una extensión de los 12-Factor App de Heroku.
Se analizan aquí los factores que impactan directamente en las pruebas.

FACTOR 1 — CODEBASE (Una base de código)
  Estado: ✓ Git con ramas feature/auth → feature/purchases → feature/sales
  Tests: ✓ Los tests viven en el mismo repositorio que el código
  Gap: ✗ No hay pipeline CI/CD que ejecute los tests automáticamente en cada push

FACTOR 4 — BACKING SERVICES (Servicios de apoyo como recursos adjuntos)
  Estado: PostgreSQL como backing service
  Tests: ✓ SaleOrderConcurrencyTest usa PostgreSQL real
  Gap: ✗ No hay perfil de test separado (application-test.yaml).
       Si la configuración de application.yaml cambia (usuario, password, host),
       los tests @SpringBootTest fallan silenciosamente o usan config incorrecta.
  IMPLEMENTAR:
    src/test/resources/application-test.yaml con:
      spring.datasource.url: jdbc:postgresql://localhost:5432/almacen_db_test
      spring.jpa.hibernate.ddl-auto: create-drop  ← genera esquema limpio por test

FACTOR 6 — PROCESSES (Procesos sin estado)
  Estado: ✓ La app es stateless (JWT, sin HttpSession)
  Tests: Los tests unitarios son stateless por diseño
  Gap: ✗ SaleOrderConcurrencyTest deja datos en la BD del dev environment.
       @AfterEach limpia, pero si el test falla antes del cleanup, los datos
       quedan huérfanos y pueden afectar runs posteriores.
  IMPLEMENTAR: @Transactional en tests que no prueban concurrencia, o usar
               una BD de test separada (application-test.yaml).

FACTOR 10 — DEV/PROD PARITY (Paridad entre entornos)
  Estado: Parcialmente cumplido
  Tests: ✓ Tests de concurrencia usan PostgreSQL real (no H2)
  Gap: ✗ Los @DataJpaTest pendientes usarían H2 por defecto, que no soporta
       exactamente la misma sintaxis que PostgreSQL (IDENTITY, CHECK, funciones).
  IMPLEMENTAR: Usar @AutoConfigureTestDatabase(replace=Replace.NONE) en todos
               los @DataJpaTest para garantizar que usan PostgreSQL real.

FACTOR 11 — LOGS (Tratar los logs como flujos de eventos)
  Estado: Spring Boot usa SLF4J + Logback — estándar correcto
  Tests: ✗ Ningún test verifica que eventos críticos se loguean correctamente.
  Gap: No hay tests para: log de aprobación de orden, log de entrega,
       log de error de concurrencia, log de acceso no autorizado.
  PRIORIDAD: Baja — no es bloqueante para funcionalidad, sí para auditoría en prod.

FACTOR 15 (extensión) — TELEMETRÍA Y OBSERVABILIDAD
  Estado: No implementado
  Tests: ✗ Sin métricas de cobertura (JaCoCo), sin dashboards de test
  IMPLEMENTAR: JaCoCo como primer paso de observabilidad del sistema de pruebas.

FACTOR 9 — DISPOSABILIDAD (Startup y shutdown rápidos)
  Tests: ✗ Los tests E2E (scripts curl) no son idempotentes — si se ejecutan
         dos veces sin limpiar la BD, los tests de creación fallan por nombres
         duplicados.
  IMPLEMENTAR: Usar sufijos únicos por ejecución (como hace SaleOrderConcurrencyTest)
               o un @BeforeAll que limpie la BD de test.

3.2.2  PASO 1 — JaCoCo: MEDICIÓN DE COBERTURA
────────────────────────────────────────────────────────────────────────────────

POR QUÉ ES NECESARIO:
  Sin métricas de cobertura no es posible saber qué porcentaje del código se
  ejecuta durante los tests, qué ramas condicionales nunca se ejercitan, ni qué
  métodos nunca son alcanzados. Los 193 tests parecían completos, pero sin números
  no había certeza. Adicionalmente, sin un umbral configurado en el pipeline, una
  regresión que redujera la cobertura pasaría desapercibida.

IMPLEMENTACIÓN:
  Se agregó el plugin jacoco-maven-plugin v0.8.11 en pom.xml con tres ejecuciones:
  1. prepare-agent: instrumenta el bytecode antes de ejecutar los tests
  2. report (fase test): genera reporte HTML en target/site/jacoco/index.html
  3. check (fase verify): falla el build si la cobertura cae bajo el umbral

  Umbral configurado: 70% de cobertura de líneas por paquete.

BUG ENCONTRADO — Falsa alarma por código auto-generado:
  Primera ejecución de ./mvnw verify falló con:
    "Rule violated for package modules.auth.mapper: lines covered ratio is 0.00"

  Causa: JaCoCo instrumenta TODAS las clases del classpath, incluyendo las
  implementaciones generadas por MapStruct en tiempo de compilación
  (UserMapperImpl, ProductMapperImpl, etc.). Estas clases existen en
  target/generated-sources y no son código que escribimos. Como los tests
  Tipo A mockean los mappers, el código MapStruct generado nunca se ejecuta,
  resultando en 0% de cobertura para esos paquetes.

  El caso extremo: auth/mapper solo contenía UserMapperImpl (generado). Al
  excluir esa clase, el paquete quedó con 0 clases, y JaCoCo interpretó
  0/0 como 0.0% (fallo del umbral).

CORRECCIÓN:
  Se actualizó la configuración del check con dos niveles de exclusión:

  a) Exclusión global de clases (aplica al reporte Y al check):
     <exclude>**/*MapperImpl.class</exclude>
     <exclude>**/dto/**</exclude>
     <exclude>**/model/**</exclude>
     <exclude>**/AlmacenesApplication.class</exclude>

  b) Exclusión de paquetes en la regla del check (paquetes que quedan vacíos
     después de las exclusiones globales):
     <excludes>
       <exclude>com.codigo2enter.almacenes</exclude>
       <exclude>com.codigo2enter.almacenes.modules.auth.mapper</exclude>
       <exclude>com.codigo2enter.almacenes.modules.inventory.mapper</exclude>
       <exclude>com.codigo2enter.almacenes.modules.purchases.mapper</exclude>
       <exclude>com.codigo2enter.almacenes.modules.sales.mapper</exclude>
     </excludes>

BUG ENCONTRADO — JwtAuthenticationFilter sin manejo de excepciones:
  El test recursos_conTokenManipulado_retorna403 falló con:
    "com.codigo2enter.almacenes.core.security.SecurityFilterTest
     recursos_conTokenManipulado_retorna403 → Runtime
     JWT signature does not match"

  Causa: JwtAuthenticationFilter llamaba a jwtUtils.extractUsername(token) sin
  try-catch. Cuando el mock lanzaba RuntimeException (simulando firma inválida),
  la excepción propagaba hacia arriba por la cadena de filtros en lugar de ser
  capturada. El resultado era un error 500 (o comportamiento indefinido de Tomcat)
  en lugar del 403 esperado. Este era un BUG REAL EN PRODUCCIÓN — cualquier
  cliente que enviara un token malformado provocaría un error del servidor.

CORRECCIÓN EN CÓDIGO DE PRODUCCIÓN (JwtAuthenticationFilter.java):
  ANTES:
    String username = jwtUtils.extractUsername(token);
    // Sin manejo de excepción — la propagación hacia arriba es un bug

  DESPUÉS:
    String username;
    try {
        username = jwtUtils.extractUsername(token);
    } catch (Exception e) {
        // Token malformado o con firma inválida — pasar sin autenticar.
        // Spring Security rechazará el request con 403 al llegar a la ruta protegida.
        filterChain.doFilter(request, response);
        return;
    }

  Justificación: el filtro debe manejar tokens inválidos gracefulmente.
  Propagarlos como excepción es incorrecto — resulta en un error 500 del servidor
  que no aporta información al cliente legítimo y puede ser explotado para
  reconocimiento de errores.

NOTA SOBRE LA ASERCIÓN "NO ES 403":
  MockMvc no tiene un método status().isNot(403). Se usó un ResultMatcher
  personalizado basado en assertNotEquals():
    .andExpect(result -> assertNotEquals(
        HttpStatus.FORBIDDEN.value(),
        result.getResponse().getStatus(),
        "Spring Security no debe bloquear POST /auth/register"))

  Esta es la forma correcta en Spring MVC Test Framework cuando se necesita
  verificar la ausencia de un código HTTP en lugar de su presencia.

3.2.9  LECCIONES APRENDIDAS
────────────────────────────────────────────────────────────────────────────────

LECCIÓN 1 — Los tests con filtros deshabilitados tienen un punto ciego crítico
  Durante 193 tests, ninguno probó que Spring Security funcionaba. El primer
  test con filtros activos encontró un bug real. La práctica de usar
  addFilters=false es válida para tests de lógica del controlador, pero debe
  complementarse con al menos una clase de tests con filtros activos.
  Aplicar en todos los proyectos Spring Security desde el inicio.

LECCIÓN 2 — Los tests encuentran bugs, no los crean
  El bug de JwtAuthenticationFilter existía desde la implementación inicial.
  Lo que ocurrió no fue que el test "estaba mal" — fue que el test encontró
  un bug que había pasado desapercibido. Cuando un test nuevo hace fallar
  el build, la primera pregunta es "¿el código tiene un bug?" antes de
  "¿el test está mal configurado?".

LECCIÓN 3 — La cobertura de ramas es la métrica más honesta
  82.4% de líneas parece bueno, pero 58.7% de ramas revela que casi la mitad
  de las condiciones if/else nunca se ejercitan. Una línea cubierta no
  significa que todos sus caminos de ejecución estén probados. La cobertura
  de ramas es más difícil de alcanzar y más valiosa como indicador de calidad.

LECCIÓN 4 — Los tests @SpringBootTest son los únicos que detectan bugs de integración
  Los 33 tests nuevos confirmaron los 3 bugs históricos ya documentados, pero ahora
  son parte del pipeline automático. Cada vez que se ejecute ./mvnw test, se verifica
  que esos bugs no regresaron. Sin @SpringBootTest, dependeríamos eternamente de
  pruebas manuales con curl para detectar regresiones de ese tipo.

LECCIÓN 5 — La configuración de herramientas también puede tener bugs
  La configuración inicial de JaCoCo falló en el primer intento de ./mvnw verify.
  Las herramientas de calidad también requieren prueba y corrección. El proceso
  correcto es: configurar → ejecutar → analizar fallos → corregir configuración →
  ejecutar nuevamente. No asumir que la primera configuración es correcta.

LECCIÓN 6 — El orden de escritura importa: tests antes de código (TDD)
  Cuando los tests se escriben después del código, frecuentemente revelan
  suposiciones implícitas que requieren correcciones. En este ciclo se corrigieron
  el try-catch del filtro JWT y la configuración de JaCoCo. Con TDD, estos
  habrían sido parte del diseño original. Escribir tests primero obliga a pensar
  en los casos de error antes de implementarlos.

PARTE 4 — GLOSARIO DE TÉRMINOS
================================================================================

@DataJpaTest
  Anotación de Spring Boot Test que levanta solo el contexto JPA (Hibernate +
  datasource). No levanta Spring MVC ni servicios. Ideal para probar
  repositorios y queries JPQL en aislamiento.

@SpringBootTest
  Levanta el contexto completo de Spring Boot. Con webEnvironment=RANDOM_PORT
  además inicia un servidor HTTP real en un puerto aleatorio.

@WebMvcTest
  Levanta solo la capa web (Spring MVC + filtros de seguridad). Los servicios
  y repositorios deben ser @MockBean.

@AutoConfigureMockMvc(addFilters = false)
  Deshabilita todos los filtros del servlet, incluyendo los de Spring Security.
  Permite probar controladores sin autenticación, pero elimina la cobertura
  de seguridad.

Optimistic Locking / @Version
  Mecanismo de control de concurrencia que usa un campo de versión para detectar
  modificaciones concurrentes. Si dos transacciones intentan actualizar el mismo
  registro, la segunda falla con ObjectOptimisticLockingFailureException.

saveAndFlush()
  Variante de JpaRepository.save() que fuerza el SQL UPDATE inmediatamente
  (sin esperar al commit de la transacción). Necesario cuando se quiere capturar
  ObjectOptimisticLockingFailureException dentro de un try-catch.

CountDownLatch
  Primitiva de sincronización de Java. Se usa para hacer que múltiples threads
  esperen hasta un punto de inicio común (startGun.countDown()) — garantiza
  solapamiento real en tests de concurrencia.

15-Factor App
  Extensión de los 12-Factor App de Heroku. Define principios para aplicaciones
  modernas cloud-native. Los 3 factores adicionales son: telemetría, principio
  de menor sorpresa (API design), y autenticación/autorización.

JaCoCo
  Java Code Coverage. Herramienta estándar para medir qué porcentaje del código
  Java se ejecuta durante los tests. Se integra con Maven como plugin y genera
  reportes HTML y XML.

JPQL
  Java Persistence Query Language. Lenguaje de consulta de JPA que opera sobre
  entidades Java en lugar de tablas SQL. Las queries @Query en los repositorios
  usan JPQL (o SQL nativo con nativeQuery=true).

Smoke Test
  Prueba mínima que verifica que el sistema arranca y está operativo.
  AlmacenesApplicationTests es el smoke test de esta aplicación.

UnnecessaryStubbingException
  Error de Mockito cuando un when().thenReturn() está configurado pero el método
  mockeado nunca se llama durante el test. Indica que el test probablemente
  cambió su flujo y hay configuración obsoleta.

Double-booking
  En el contexto de inventario: reservar más unidades de las que físicamente
  existen. El @Version (Optimistic Locking) previene el double-booking cuando
  múltiples transacciones intentan reservar stock simultáneamente.
