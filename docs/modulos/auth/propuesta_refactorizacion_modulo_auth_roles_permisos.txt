1.2  CÓDIGO JAVA

JwtAuthenticationFilter:
  - Extrae username del token → correcto.
  - NO extrae roles del token → los carga como lista vacía (Collections.emptyList()).
  - Consecuencia: `Authentication.getAuthorities()` siempre vacío → hasRole() siempre falla.

  - Única regla: anyRequest().authenticated() → sin distinción por rol.
  - Sin @EnableMethodSecurity → no se pueden usar @PreAuthorize en métodos.

  - registerUser(): público, asigna ROLE_WAREHOUSEMAN a cualquier nuevo usuario.
  - login(): correcto.
  - Sin gestión de usuarios (no hay CRUD de usuarios, ni cambio de roles).

UserController:
  - Solo POST /auth/register y POST /auth/login.
  - Sin endpoints de gestión de usuarios.

1.3  BRECHA CRÍTICA
-------------------
El sistema no puede diferenciar qué operaciones puede hacer cada usuario.
Actualmente cualquier usuario autenticado puede:
  - Aprobar órdenes de compra (debería requerir MANAGER o ADMIN)
  - Crear productos (debería requerir MANAGER o ADMIN)
  - Cancelar ventas (debería requerir control)
  - Gestionar usuarios (no existe la funcionalidad)

2. DECISIONES DE DISEÑO FUNDAMENTALES
================================================================================

DECISIÓN 1 — Registro de usuarios: de público a administrado
  ANTES: cualquiera puede registrarse en POST /auth/register → obtiene ROLE_WAREHOUSEMAN.
  DESPUÉS: solo ADMIN puede crear usuarios con el rol que corresponda.
  JUSTIFICACIÓN: un sistema de gestión de almacenes de producción no debe permitir
  autoregistro. El inventario, las ventas y las compras son operaciones críticas
  de negocio — el acceso debe ser administrado explícitamente.
  MIGRACIÓN: POST /auth/register se convierte en POST /api/v1/auth/users (ADMIN only).
  El endpoint de login permanece público.

DECISIÓN 2 — Estrategia de autorización: URL-based en SecurityConfig
  Se usarán reglas URL en SecurityConfig.authorizeHttpRequests() en lugar de
  @PreAuthorize en métodos de servicio/controlador.
  JUSTIFICACIÓN: las reglas centralizadas en SecurityConfig son auditables en
  un solo lugar, más fáciles de testear con SecurityFilterTest (Tipo B*), y
  no contaminan el código de negocio con anotaciones de seguridad.
  @PreAuthorize se reserva para casos que no pueden expresarse con patrones URL.

DECISIÓN 3 — Roles extraídos del JWT, no consultados en BD por request
  El JwtAuthenticationFilter extrae los roles del claim 'roles' del token
  y los convierte a GrantedAuthority. No se hace una consulta a users/roles
  por cada request.
  JUSTIFICACIÓN: el sistema es stateless (SessionCreationPolicy.STATELESS).
  Consultar la BD en cada request afecta la latencia y no es compatible con
  la arquitectura JWT. Si los roles de un usuario cambian, debe iniciar sesión
  de nuevo para obtener un token actualizado — comportamiento estándar en JWT.

DECISIÓN 4 — Soft delete para usuarios (active=false)
  Los usuarios no se eliminan físicamente. La columna `active` ya existe en `users`.
  JUSTIFICACIÓN: los usuarios tienen FK en created_by/updated_by de todos los
  módulos. Una eliminación física violaría la integridad referencial. El soft
  delete preserva el historial de auditoría.

DECISIÓN 5 — users.updated_at (sin updated_by)
  Se agrega updated_at a la tabla users. No se agrega updated_by.
  JUSTIFICACIÓN: updated_by crearía una FK circular (users → users). Para
  registrar quién modificó a quién, se puede usar el log de auditoría del
  sistema operativo o una tabla de auditoría separada en el futuro.

│ Rol                 │ Descripción y responsabilidad                        │

│                     │ Acceso completo a todos los módulos. Puede hacer todo │

│                     │ de usuarios.                                          │

│ ROLE_MANAGER        │ Gerente de almacén. Gestiona el catálogo (categorías, │
│                     │ productos, proveedores), aprueba órdenes de compra y  │
│                     │ ventas. No puede gestionar usuarios.                  │

│ ROLE_WAREHOUSEMAN   │ Operador de almacén. Registra movimientos de stock,  │

│                     │ No puede aprobar órdenes ni modificar el catálogo.   │

│                     │ Puede crear/cancelar sus propias órdenes. No tiene   │
│                     │ acceso a compras ni a modificar inventario.           │

-- PASO 5.3 — Asignar ROLE_ADMIN al usuario tester01

-- Se le agrega ROLE_ADMIN para que pueda gestionar el sistema.
-- Se mantiene ROLE_WAREHOUSEMAN para que siga teniendo acceso a las
-- operaciones de bodeguero.
-- ─────────────────────────────────────────────────────────────────────────
INSERT INTO user_roles (user_id, role_id)

FROM users u, roles r
WHERE u.username = 'tester01'
  AND r.name = 'ROLE_ADMIN'
  AND NOT EXISTS (
    SELECT 1 FROM user_roles ur
    WHERE ur.user_id = u.id AND ur.role_id = r.id
  );

-- ─────────────────────────────────────────────────────────────────────────
-- PASO 5.4 — Verificar estado post-migración
-- ─────────────────────────────────────────────────────────────────────────
SELECT u.username, STRING_AGG(r.name, ', ') AS roles
FROM users u
JOIN user_roles ur ON ur.user_id = u.id
JOIN roles r ON r.id = ur.role_id
GROUP BY u.username;

CAMBIO PRINCIPAL: extraer roles del claim 'roles' y cargarlos como GrantedAuthority.

CÓDIGO ACTUAL:

  List<SimpleGrantedAuthority> authorities = jwtUtils.extractRoles(token)
          .stream()
          .map(SimpleGrantedAuthority::new)
          .collect(Collectors.toList());

  UsernamePasswordAuthenticationToken authToken = new UsernamePasswordAuthenticationToken(
      username,
      null,
      authorities ← roles cargados desde el JWT
  );

NUEVO MÉTODO en JwtUtils — extractRoles():
  @SuppressWarnings("unchecked")
  public List<String> extractRoles(String token) {
      Claims claims = parseClaims(token);
      Object rolesObj = claims.get("roles");
      if (rolesObj instanceof List<?> list) {
          return list.stream()
                  .filter(String.class::isInstance)
                  .map(String.class::cast)
                  .collect(Collectors.toList());
      }
      return List.of();
  }

IMPORTS adicionales en JwtAuthenticationFilter:

      .requestMatchers(HttpMethod.DELETE,

      // ── SALES: transiciones de estado ─────────────────────────────────
      // deliver: ADMIN, MANAGER, WAREHOUSEMAN
      .requestMatchers("/api/v1/sales/orders/*/deliver").hasAnyRole("ADMIN","MANAGER","WAREHOUSEMAN")
      // approve/cancel: ADMIN, MANAGER, SALES (SALES solo puede cancelar propias)
      .requestMatchers(HttpMethod.PATCH,
          "/api/v1/sales/orders/**").hasAnyRole("ADMIN","MANAGER","SALES")

      // ── CUALQUIER OTRA RUTA: autenticado ──────────────────────────────
      .anyRequest().authenticated()
  )

NOTA IMPORTANTE sobre el orden de las reglas:
  Spring Security evalúa las reglas en el orden en que se definen.
  La primera que coincide gana. Las reglas más específicas (paths con *) deben
  ir ANTES de las más generales. Por ejemplo:
    - "/api/v1/purchases/orders/*/receive" antes de "/api/v1/purchases/orders/**"
    - DELETE /inventory/products/** (ADMIN only) antes del GET /inventory/** (todos)

  UserResponseDTO getMyProfile();
  void changePassword(ChangePasswordDTO dto);

  // registerUser() y login() se mantienen pero registerUser() quedará obsoleto
  // (el controlador ya no lo expone públicamente).

IMPLEMENTACIÓN — lógica relevante por método:

  createUser(UserCreateDTO dto):
    1. Validar unicidad de username (existsByUsername)
    2. Validar unicidad de email (existsByEmail)
    3. Validar que cada rol en dto.getRoles() existe en RoleRepository
    4. Cifrar contraseña con BCrypt
    5. Crear User con active=true, createdAt=now()
    6. Asignar roles encontrados
    7. Persistir y retornar UserResponseDTO

  updateUser(Long id, UserUpdateDTO dto):
    1. findUserOrThrow(id) — verifica que existe y está activo
    2. Validar que nuevo username no pertenezca a otro usuario diferente
    3. Validar que nuevo email no pertenezca a otro usuario diferente
    4. Actualizar campos escalares
    5. updatedAt = now()
    6. Hibernate dirty-checking persiste (sin save() explícito)

  deactivateUser(Long id):
    1. findUserOrThrow(id)
    2. Verificar que no se está desactivando al propio usuario autenticado
       (el ADMIN no puede suicidarse del sistema sin un segundo ADMIN)
    3. active = false, updatedAt = now()
    4. Hibernate dirty-checking persiste

  assignRoles(Long id, UserRoleAssignDTO dto):
    1. findUserOrThrow(id)
    2. Validar que cada rol en dto.getRoles() existe en RoleRepository
    3. user.getRoles().clear()  ← vaciar roles actuales
    4. Agregar los nuevos roles
    5. updatedAt = now()
    6. save() explícito — la relación @ManyToMany requiere save() para
       que Hibernate sincronice la tabla user_roles

  getMyProfile():
    1. resolveAuthenticatedUser()
    2. Retornar UserResponseDTO sin password

  changePassword(ChangePasswordDTO dto):
    1. resolveAuthenticatedUser()
    2. passwordEncoder.matches(dto.getCurrentPassword(), user.getPassword())
       → si false, RuntimeException("Contraseña actual incorrecta.")
    3. Validar dto.getNewPassword().equals(dto.getConfirmPassword())
       → si false, RuntimeException("Las contraseñas nuevas no coinciden.")
    4. user.setPassword(passwordEncoder.encode(dto.getNewPassword()))
    5. updatedAt = now()
    6. save() explícito

  MÉTODO PRIVADO findUserOrThrow(Long id):
    return userRepository.findById(id)
            .filter(User::isActive)
            .orElseThrow(() -> new RuntimeException(
                "Usuario con id " + id + " no encontrado o inactivo."));

10. RIESGOS Y MITIGACIONES
================================================================================

RIESGO 1 — Orden de reglas en SecurityConfig causa conflictos
  Probabilidad: alta (Spring Security evalúa la primera regla que coincide).
  Síntoma: DELETE /inventory/products/{id} (ADMIN only) y
           GET /inventory/products/{id} (todos) — el DELETE podría ser
           bloqueado si la regla de DELETE está después de una regla
           demasiado general.
  Mitigación: ordenar siempre de más específico a más general. Agregar
              test SecurityFilterTest para cada caso límite.

RIESGO 2 — Token con roles obsoletos (usuario cuyo rol cambió)
  Probabilidad: baja en desarrollo, real en producción.
  Síntoma: usuario cuyo rol fue quitado sigue teniendo acceso durante
           las 2 horas de vigencia del token.
  Mitigación en esta iteración: documentar que cambiar roles requiere
                                nuevo login. No implementar revocación
                                de tokens (complejidad desproporcionada
                                para esta fase).
  Mitigación futura: lista negra de tokens (Redis) o tokens de vida corta
                     (15 min) con refresh tokens.

RIESGO 3 — tester01 pierde acceso durante la migración
  Probabilidad: media (si la migración SQL falla parcialmente).
  Síntoma: tester01 tiene ROLE_WAREHOUSEMAN pero los nuevos endpoints
           requieren ROLE_ADMIN.
  Mitigación: ejecutar el INSERT de ROLE_ADMIN a tester01 ANTES de
              desplegar el código nuevo. Verificar con SELECT antes
              de reiniciar la app.

RIESGO 4 — Tests existentes (Tipo B, addFilters=false) siguen pasando
           aunque la lógica de negocio sea incorrecta
  Probabilidad: existente desde siempre.
  Síntoma: un test de controlador pasa aunque el endpoint real rechace
           la petición por falta de rol.
  Mitigación: los tests Tipo B* de SecurityFilterTest cubren este gap.
              Los tests Tipo C (@SpringBootTest) con flujos de múltiples
              roles cubren el escenario end-to-end.

RIESGO 5 — Regresión en módulos existentes por las nuevas reglas de SecurityConfig
  Probabilidad: media (cualquier test Tipo C que envíe requests sin el rol correcto).
  Síntoma: AuditAndConstraintIntegrationTest o SaleOrderConcurrencyTest
           fallan con 403 porque el token usado no tiene el rol necesario.
  Mitigación: los tests Tipo C usan tester01 (que tendrá ROLE_ADMIN tras la
              migración) → tendrá acceso a todo. No requiere cambios en
              los tests existentes.
