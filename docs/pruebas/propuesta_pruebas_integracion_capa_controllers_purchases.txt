JUSTIFICACIÓN
─────────────
Los tests unitarios de los servicios (SupplierServiceImplTest con 13 tests y
PurchaseOrderServiceImplTest con 29 tests) cubren exhaustivamente la lógica de
negocio. Sin embargo, existen aspectos que SOLO pueden verificarse en la capa
del controlador:

  1. Enrutamiento HTTP correcto
     ¿El método anotado con @PatchMapping("/{id}/approve") realmente responde
     a PATCH y no a PUT? ¿La ruta /supplier/{supplierId}/status/{status} no
     colisiona con /supplier/{supplierId}? Los tests unitarios de servicios
     no ejecutan el DispatcherServlet — solo la capa de integración web lo hace.

  2. @Valid está colocado en el parámetro correcto
     Es el bug más común en controladores: olvidar @Valid en el @RequestBody
     hace que Jakarta ignore todas las anotaciones del DTO y el servicio recibe
     datos inválidos. Un test que envíe body inválido y espere HTTP 400 detecta
     esto inmediatamente.

  3. Serialización JSON correcta
     ¿Jackson serializa BigDecimal como número y no como String?
     ¿LocalDateTime sigue el formato esperado? ¿Los campos null aparecen
     como null y no se omiten? Los tests de servicio usan mocks —
     nunca pasan por el serializador real.

  4. Códigos HTTP semánticamente correctos
     ¿createOrder retorna 201 y no 200? ¿removeDetail retorna 204 y no 200?
     Estos detalles son críticos para el contrato con Angular y solo se
     verifican en la capa web.

  5. Endpoints sin body son invocables sin error
     Los PATCH (approve, receive, cancel) no reciben body. Es válido verificar
     que Spring no exige body donde no corresponde.


CRITERIO DE SELECCIÓN DE TESTS
───────────────────────────────
Se seleccionaron tests con el siguiente criterio de valor/esfuerzo:

  ✓ INCLUIR:
    · Happy path de cada endpoint → verifica enrutamiento + status HTTP + estructura JSON
    · Tests de validación (@Valid) en cada endpoint con @RequestBody
      → detectan el bug de "olvidar @Valid"
    · Tests de PATCH sin body → verifican que no se exige body donde no corresponde

  ✗ EXCLUIR:
    · Lógica de negocio (proveedor no encontrado, orden no en PENDING, etc.)
      → ya cubierta exhaustivamente en los 42 tests de servicios
    · Casos edge del servicio (stock insuficiente, máquina de estados)
      → responsabilidad del servicio, no del controlador
    · Tests de autorización JWT
      → addFilters=false los desactiva; se probarían en tests E2E

  Resultado: tests concisos, sin duplicación con la capa de servicios,
  enfocados en lo que SOLO el controlador puede validar.


CONFIGURACIÓN BASE
──────────────────
  @WebMvcTest(XxxController.class)        → contexto web reducido
  @AutoConfigureMockMvc(addFilters=false) → desactiva filtros JWT
  @Autowired MockMvc mockMvc
  @MockBean XxxService                    → aísla el controlador
  @MockBean JwtUtils                      → satisface dependencia de SecurityConfig

  Patrón ACT+ASSERT (sin ARRANGE en tests de validación):
    mockMvc.perform(...).andExpect(status().isXxx()).andExpect(jsonPath(...))

  Tests de mayor valor:
  ─────────────────────

  TEST 7 (GET /supplier/{supplierId}/status/{status}):
    Es el endpoint con la ruta más compleja — dos PathVariables en un path
    anidado. Un error en la anotación @GetMapping o en la firma del método
    (@PathVariable mal nombrado) haría que Spring no encuentre el endpoint.
    Este test garantiza que el enrutamiento doble funciona correctamente.

  TEST 2 (POST / details vacíos) y TEST 3 (supplierId nulo):
    Los dos casos de validación más críticos del endpoint principal. Si @Valid
    falta en el PurchaseOrderRequestDTO, órdenes con datos inválidos llegarían
    al servicio y el error resultante sería confuso. Estos tests son la red
    de seguridad más barata para ese escenario.

  TEST 11, 12, 13 (PATCH sin body):
    Verifican que los endpoints de transición de estado no exigen body donde
    no corresponde. Es común que frameworks rechacen PATCH sin Content-Type
    o sin body — este test garantiza que Spring los acepta correctamente.

  TEST 15 (POST /details con quantity=0) y TEST 17 (PUT /details con unitPrice=0.00):
    Las validaciones de los subrecursos son igualmente importantes que las
    de la orden. Un quantity=0 llegaría al servicio y produciría un error
    de negocio en lugar del 400 más informativo de Jakarta.

  Criterio final de selección:
  ─────────────────────────────
  Cada test de esta suite responde a la pregunta:
    "¿Podría este error pasar inadvertido en los tests de servicio?"
  Si la respuesta es Sí → el test pertenece a la capa del controlador.
  Si la respuesta es No → ya está cubierto y se excluye para evitar duplicación.
