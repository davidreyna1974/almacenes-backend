  1. CONTEXTO Y JUSTIFICACIÓN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Módulos implementados hasta este punto:
    - auth       → Autenticación, autorización, RBAC (4 roles)
    - inventory  → Categorías, productos, movimientos de stock (Kardex)
    - purchases  → Proveedores, órdenes de compra (PENDING→APPROVED→RECEIVED)
    - sales      → Clientes, órdenes de venta (PENDING→APPROVED→DELIVERED)

  El módulo `reports` es el siguiente paso natural porque:

  a) La data necesaria ya existe en la base de datos y no requiere nuevas tablas.
     El módulo sales capturó `unitCost` en cada `sale_order_detail` específicamente
     para habilitar análisis financiero futuro. Ese futuro es `reports`.

  b) Sin reportes, los datos de los módulos anteriores solo son consultables
     registro por registro vía los endpoints de cada módulo. No existe una vista
     agregada que permita tomar decisiones de negocio.

  c) Un sistema de gestión de almacenes (WMS) sin reportes no cumple los
     estándares mínimos de la industria. Los reportes son la capa de inteligencia
     que convierte datos operativos en información para la toma de decisiones.

  Problemática que resuelve:
    - El dueño no puede saber si su negocio es rentable sin calcular manualmente
      revenue y costos desde múltiples tablas.
    - La dirección no puede identificar qué productos generan más valor ni cuáles
      tienen baja rotación.
    - El personal operativo no tiene una vista consolidada de lo que requiere
      atención inmediata (stock bajo, órdenes pendientes).

  2. ANÁLISIS DE AUDIENCIAS Y NECESIDADES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  La diferencia entre audiencias no es solo qué datos ver, sino qué pregunta
  responden y con qué frecuencia los consultan.

  ┌──────────────────┬─────────────────────────────────┬──────────────┬──────────────────┐
  │ Audiencia        │ Pregunta central                │ Granularidad │ Frecuencia       │
  ├──────────────────┼─────────────────────────────────┼──────────────┼──────────────────┤
  │ Dueño            │ ¿Es rentable mi negocio?        │ Totales y %  │ Semanal/mensual  │
  │ Alta dirección   │ ¿Qué productos/proveedores      │ Por categoría│ Diario/semanal   │
  │                  │ rinden más?                     │ y proveedor  │                  │
  │ Personal         │ ¿Qué necesita atención hoy?     │ Por producto │ Varias veces     │
  │ operativo        │                                 │ individual   │ al día           │
  └──────────────────┴─────────────────────────────────┴──────────────┴──────────────────┘

  Roles del sistema que corresponden a cada audiencia:
    - Dueño          → ROLE_ADMIN (visión completa de todos los reportes)
    - Alta dirección → ROLE_ADMIN + ROLE_MANAGER
    - Operativo      → ROLE_WAREHOUSEMAN + ROLE_SALES (reportes operativos)

  Criterio de diseño aplicado:
    Los reportes del dueño muestran totales y porcentajes. No necesita ver el
    Kardex de un producto específico para tomar decisiones estratégicas.
    El operativo necesita el Kardex y los alertas de stock, pero no la
    rentabilidad global ni el análisis ABC. Esta separación evita sobrecargar
    cada interfaz con información irrelevante para su rol.

  3. CATÁLOGO DE REPORTES POR AUDIENCIA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ── 3.1 DUEÑO — Visión estratégica ──────────────────────────────────────────

  Reporte 1: Dashboard Ejecutivo
    Pregunta: ¿Cómo está el negocio en este momento?
    Datos fuente: SaleOrder(DELIVERED), SaleOrderDetail, Product
    Campos de respuesta:
      - totalRevenue        → SUM(sale_order_details.subtotal) de órdenes DELIVERED
      - totalCogs           → SUM(sod.quantity × sod.unit_cost) de órdenes DELIVERED
                              (excluye detalles con unitCost NULL — indicado en respuesta)
      - grossMargin         → totalRevenue − totalCogs
      - grossMarginPct      → (grossMargin / totalRevenue) × 100
      - inventoryValue      → SUM(current_stock × unit_cost) de productos activos
      - pendingPurchaseOrders → COUNT de PurchaseOrders en PENDING o APPROVED
      - pendingSaleOrders    → COUNT de SaleOrders en PENDING o APPROVED
      - cogsCompleteness    → % de detalles de venta con unitCost definido
      - generatedAt         → timestamp del momento de cálculo
    Parámetros opcionales: from, to (rango de fechas para revenue y COGS)
    Acceso: ADMIN

  Reporte 2: Valuación de Inventario
    Pregunta: ¿Cuánto vale mi inventario actual en costo?
    Datos fuente: Product, Category
    Campos de respuesta:
      - totalValue          → SUM total de todos los productos activos
      - categories          → lista de:
          categoryId, categoryName, productCount,
          categoryValue (SUM del grupo), pct (% del total)
      - productsWithoutCost → COUNT de productos activos sin unitCost definido
      - generatedAt
    Parámetros: ninguno (siempre refleja el estado actual del inventario)
    Acceso: ADMIN, MANAGER

  Reporte 3: Rentabilidad por Período
    Pregunta: ¿Cuánto gané (o perdí) en un período?
    Datos fuente: SaleOrder(DELIVERED), SaleOrderDetail
    Campos de respuesta:
      - from, to            → período consultado
      - totalRevenue
      - totalCogs
      - grossMargin
      - grossMarginPct
      - deliveredOrderCount → número de órdenes entregadas en el período
      - avgTicket           → totalRevenue / deliveredOrderCount
      - cogsCompleteness    → % de líneas de venta con costo definido
      - generatedAt
    Parámetros: from, to (obligatorios para este reporte)
    Acceso: ADMIN, MANAGER

  ── 3.2 ALTA DIRECCIÓN — Visión táctica ─────────────────────────────────────

  Reporte 4: Top Productos
    Pregunta: ¿Qué productos generan más valor?
    Datos fuente: SaleOrderDetail + SaleOrder(DELIVERED) + Product + Category
    Campos de respuesta (lista ordenada por totalRevenue DESC):
      - rank
      - productId, sku, name, categoryName
      - totalQuantitySold   → SUM(quantity)
      - totalRevenue        → SUM(subtotal)
      - totalCogs           → SUM(quantity × unitCost) donde unitCost NOT NULL
      - grossMargin         → totalRevenue − totalCogs
      - grossMarginPct
    Parámetros: from, to (opcional), limit (default 10, max 50)
    Acceso: ADMIN, MANAGER

  Reporte 5: Análisis ABC
    Pregunta: ¿Cuáles son mis productos más valiosos y cuáles tienen poco impacto?
    Fundamento: Principio de Pareto aplicado a inventario.
                Categoría A = 80% del revenue (típicamente 20% de los productos)
                Categoría B = siguiente 15% del revenue
                Categoría C = 5% restante del revenue
    Datos fuente: SaleOrderDetail + SaleOrder(DELIVERED) + Product
    Campos de respuesta (lista ordenada por totalRevenue DESC):
      - classification      → "A", "B" o "C"
      - productId, sku, name
      - totalRevenue        → revenue generado por el producto en el período
      - revenuePct          → % que representa del revenue total
      - cumulativePct       → % acumulado (usado para determinar A/B/C)
    Algoritmo:
      1. Calcular totalRevenue por producto (solo DELIVERED)
      2. Ordenar DESC
      3. Calcular cumulativePct incremental
      4. Asignar A si cumulativePct <= 80, B si <= 95, C el resto
    Parámetros: from, to (opcional — default: último año)
    Nota: requiere mínimo 3 productos con ventas para clasificación significativa
    Acceso: ADMIN, MANAGER

  Reporte 6: Rotación de Inventario
    Pregunta: ¿Qué tan rápido se vende mi inventario?
    Fundamento: Rotación alta = el inventario se renueva frecuentemente (positivo).
                Rotación baja = el inventario está paralizado (capital inmovilizado).
    Fórmula: Rotación = COGS del período / Valor actual del inventario a costo
    Datos fuente: SaleOrderDetail(DELIVERED) + Product
    Campos de respuesta (lista por producto):
      - productId, sku, name, categoryName
      - cogsInPeriod        → SUM(quantity × unitCost) de órdenes DELIVERED
      - currentInventoryValue → currentStock × unitCost
      - turnoverRate        → cogsInPeriod / currentInventoryValue
                              (null si currentInventoryValue = 0 o unitCost null)
      - interpretation      → "Alta" (>4), "Media" (1-4), "Baja" (<1), "Sin datos"
    Parámetros: from, to (default: último año)
    Acceso: ADMIN, MANAGER

  Reporte 7: Análisis de Compras por Proveedor
    Pregunta: ¿Cuánto compro a cada proveedor y qué tan activos son?
    Datos fuente: PurchaseOrder(RECEIVED) + PurchaseOrderDetail + Supplier
    Campos de respuesta (lista por proveedor):
      - supplierId, supplierName, rfc
      - orderCount          → número de órdenes recibidas en el período
      - totalAmount         → SUM(totalAmount) de órdenes RECEIVED
      - avgOrderAmount      → totalAmount / orderCount
      - lastOrderDate       → fecha de la orden más reciente
    Parámetros: from, to (opcional)
    Acceso: ADMIN, MANAGER

  Reporte 8: Tendencia de Ventas
    Pregunta: ¿Cómo evolucionan mis ventas en el tiempo?
    Datos fuente: SaleOrder(DELIVERED)
    Campos de respuesta (lista de períodos):
      - period              → valor del período (ej. "2025-01", "2025-W03", "2025-01-15")
      - revenue             → SUM(totalAmount) de órdenes entregadas en el período
      - orderCount          → número de órdenes entregadas
      - avgTicket           → revenue / orderCount
    Parámetros:
      - from, to (obligatorios)
      - groupBy: "DAY" | "WEEK" | "MONTH" (default: MONTH)
    Nota técnica: el agrupamiento por período se implementa con FUNCTION('TO_CHAR', ...)
                  en JPQL, específico del dialecto PostgreSQL.
    Acceso: ADMIN, MANAGER

  ── 3.3 PERSONAL OPERATIVO — Visión operativa ────────────────────────────────

  Reporte 9: Productos Bajo Mínimo de Stock
    Pregunta: ¿Qué productos necesito reabastecer hoy?
    Datos fuente: Product + Category
    Campos de respuesta (lista ordenada por déficit DESC):
      - productId, sku, name, categoryName
      - currentStock
      - minimumStock
      - availableStock      → currentStock − reservedStock
      - reservedStock
      - deficit             → minimumStock − currentStock (siempre >= 1 en este reporte)
    Sin parámetros. Siempre refleja el estado actual.
    Acceso: ADMIN, MANAGER, WAREHOUSEMAN

  Reporte 10: Kardex por Producto
    Pregunta: ¿Cuál es el historial de movimientos de este producto?
    Diferencia con el endpoint existente GET /inventory/products/{id}/movements:
      Este reporte incluye saldo acumulado por movimiento y resumen de apertura/cierre,
      información que el endpoint de inventario no calcula.
    Datos fuente: StockMovement + Product
    Campos de respuesta:
      - productId, sku, name
      - from, to
      - openingStock        → stock antes del primer movimiento del período
                              (calculado como: stockActual − netMovements del período)
      - closingStock        → currentStock actual
      - totalIn             → SUM(quantity) de movimientos IN en el período
      - totalOut            → SUM(quantity) de movimientos OUT en el período
      - movements (lista):
          date, type (IN/OUT), quantity, reason, balance (saldo acumulado),
          createdByUsername
    Parámetros: productId (path), from, to (opcional — default: último mes)
    Acceso: ADMIN, MANAGER, WAREHOUSEMAN

  Reporte 11: Operaciones Pendientes
    Pregunta: ¿Qué órdenes requieren acción ahora mismo?
    Datos fuente: PurchaseOrder(PENDING/APPROVED) + SaleOrder(PENDING/APPROVED)
    Campos de respuesta:
      - pendingPurchaseOrders (lista):
          orderId, orderNumber, status, supplierName,
          createdAt, totalAmount, detailCount
      - pendingSaleOrders (lista):
          orderId, orderNumber, status, clientName,
          createdAt, totalAmount, detailCount
      - summary:
          totalPendingPurchases, totalPendingSales
    Sin parámetros. Siempre refleja el estado actual.
    Acceso: ADMIN, MANAGER, WAREHOUSEMAN, SALES

  Reporte 12: Resumen de Movimientos de Stock
    Pregunta: ¿Cuántas unidades entraron y salieron en un período?
    Datos fuente: StockMovement + Product + Category
    Campos de respuesta:
      - from, to
      - totalIn             → SUM(quantity) de movimientos IN
      - totalOut            → SUM(quantity) de movimientos OUT
      - netMovement         → totalIn − totalOut
      - movementsByType (lista agrupada):
          type, reason (agrupado por razón), count, totalQuantity
    Parámetros: from, to (default: hoy)
    Acceso: ADMIN, MANAGER, WAREHOUSEMAN

  4. DISPONIBILIDAD DE DATOS EN EL SCHEMA ACTUAL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Toda la información necesaria existe en tablas actuales. No se requieren
  nuevas tablas para ninguno de los 12 reportes.

  Cálculos y su origen:

    Revenue total:
      SELECT SUM(sod.subtotal)
      FROM sale_order_details sod
      JOIN sale_orders so ON sod.sale_order_id = so.id
      WHERE so.status = 'DELIVERED'
        AND so.delivered_at BETWEEN :from AND :to

    COGS (Costo de Ventas):
      SELECT SUM(sod.quantity * sod.unit_cost)
      FROM sale_order_details sod
      JOIN sale_orders so ON sod.sale_order_id = so.id
      WHERE so.status = 'DELIVERED'
        AND sod.unit_cost IS NOT NULL           -- limitación conocida
        AND so.delivered_at BETWEEN :from AND :to

    Valuación de inventario:
      SELECT SUM(p.current_stock * p.unit_cost)
      FROM products p
      WHERE p.active = true
        AND p.unit_cost IS NOT NULL

    Margen bruto:
      Revenue − COGS (cálculo en Java, no en SQL)

    Rotación:
      COGS del período / SUM(current_stock * unit_cost)

  Limitación conocida y manejo:
    `sale_order_details.unit_cost` es nullable (captura progresiva desde el módulo
    sales). Los reportes que usan COGS excluirán detalles con unit_cost NULL y
    retornarán el campo `cogsCompleteness` indicando qué porcentaje del revenue
    tiene costo asociado. Si cogsCompleteness < 100%, los valores de COGS y margen
    son aproximados — el reporte lo indicará explícitamente.

  Campos relevantes confirmados en entidades:
    Product:          price, unitCost (nullable), currentStock, minimumStock,
                      reservedStock, active, @ManyToOne Category
    SaleOrderDetail:  quantity, unitPrice, unitCost (nullable), subtotal,
                      @ManyToOne Product, @ManyToOne SaleOrder
    SaleOrder:        totalAmount, status, createdAt, deliveredAt, cancelledAt,
                      @ManyToOne Client, @OneToMany details
    StockMovement:    quantity, type (IN/OUT), reason, createdAt,
                      @ManyToOne Product, @ManyToOne createdBy
    PurchaseOrder:    totalAmount, status, createdAt, receivedAt,
                      @ManyToOne Supplier
    PurchaseOrderDetail: quantity, unitPrice, subtotal, @ManyToOne Product

  5. ARQUITECTURA TÉCNICA DEL MÓDULO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Total: 1 controller + 12 DTOs + 3 interfaces + 3 implementaciones = 19 clases

  Decisiones de diseño:

  a) Sin model/, mapper/ ni repository/ propios.
     El módulo reports consume los repositorios de otros módulos directamente
     (patrón de módulo de solo lectura). No persiste, no transforma entidades:
     solo agrega y proyecta datos existentes.

  b) 3 servicios separados por audiencia, no 1 servicio monolítico.
     Criterio: cada servicio tiene dependencias de repositorios distintas y una
     audiencia con responsabilidad bien definida. Un único servicio con los 12
     métodos tendría 7+ repositorios inyectados y 800+ líneas. La separación
     respeta el principio de responsabilidad única (SRP).

  c) Sin mappers MapStruct.
     Los DTOs de reportes son vistas agregadas construidas desde múltiples entidades
     y cálculos en Java. MapStruct opera sobre conversiones 1-a-1 de entidad→DTO.
     Las proyecciones de reportes (con SUM, GROUP BY, cálculos de ABC) se
     construyen directamente en el servicio. Patrón ya establecido en
     ReservationServiceImpl del módulo sales.

  d) Todos los métodos de servicio con @Transactional(readOnly = true).
     El módulo no escribe ningún dato. readOnly = true le comunica a Hibernate
     que puede omitir el flush y optimizar la sesión de lectura.

  e) Parámetros de fecha como LocalDate, no LocalDateTime.
     Los reportes filtran por día completo. El servicio convierte:
       from → from.atStartOfDay()
       to   → to.plusDays(1).atStartOfDay() (exclusive)
     Esto evita que ventas a las 23:59 del día final queden excluidas.

  6. ESPECIFICACIÓN DE ENDPOINTS Y CONTROL DE ACCESO RBAC

  Parámetros de fecha (query params, formato ISO 8601: YYYY-MM-DD):
    - from  (opcional según reporte — ver catálogo sección 3)
    - to    (opcional según reporte — ver catálogo sección 3)

  Parámetros adicionales:
    - /products/top-performers : limit (int, default 10, max 50)
    - /sales/trend             : groupBy (DAY | WEEK | MONTH, default MONTH)
    - /inventory/kardex/{id}   : from, to (default: último mes)

  Justificación del acceso diferenciado:
    - /dashboard/executive solo ADMIN: contiene márgenes y rentabilidad global,
      información que en una empresa el dueño puede no querer compartir con MANAGER.
    - /operations/pending para SALES: el vendedor necesita ver el estado de las
      órdenes de venta pendientes para dar seguimiento a sus clientes.
    - Kardex y movimientos NO para SALES: información de inventario interno que
      no es relevante para la función comercial.

  Lógica de negocio no trivial que va en el servicio (no en JPQL):

  Algoritmo ABC:
    1. Obtener revenueByProduct(from, to) desde repositorio
    2. Calcular totalRevenue = SUM de todos
    3. Ordenar productos por revenue DESC
    4. Iterar acumulando cumulativePct:
       cumulativePct += (productRevenue / totalRevenue) * 100
    5. Asignar: A si cumulativePct <= 80, B si <= 95, C resto
    6. Caso edge: si hay menos de 3 productos con ventas, devolver lista
       con clasificación best-effort y advertencia en el DTO

  Saldo acumulado del Kardex:
    El Kardex requiere mostrar el saldo después de cada movimiento.
    Este saldo no está almacenado — se calcula iterando los movimientos
    en orden cronológico y aplicando: saldo = anterior + IN - OUT
    El "saldo de apertura" se calcula como:
    openingStock = currentStock − SUM(IN en período) + SUM(OUT en período)
    (trabaja hacia atrás desde el estado actual)
